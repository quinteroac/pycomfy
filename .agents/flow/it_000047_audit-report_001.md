#### C-01 ‚Äî Wrong ARIA role on media type selector group

- **Location**: `frontend/src/components/MediaTypeSelector.tsx` line 17
- **Severity**: Critical
- **Category**: Accessibility
- **Description**: The container uses `role="group"` but contains `role="radio"` children. The correct container for radio buttons is `role="radiogroup"`.
- **Impact**: Assistive technologies (NVDA, VoiceOver) cannot correctly announce the semantics of the control group. Screen readers will not automatically announce it as a radio group, and keyboard navigation behaviour may be incorrect.
- **WCAG**: 4.1.2 Name, Role, Value (Level A)
- **Recommendation**: Change `role="group"` to `role="radiogroup"` on the wrapper div.
- **Suggested command**: `harden`

### High-Severity Issues

#### H-01 ‚Äî Insufficient contrast: white text on accent background

- **Location**: `frontend/src/App.css` (--accent: #8b5cf6), `ChatBubble.module.css` (.bubble-user .bubble-content), `MediaTypeSelector.module.css` (.button.active)
- **Severity**: High
- **Category**: Accessibility
- **Description**: White (#ffffff) text on the accent colour `#8b5cf6` yields a contrast ratio of approximately **4.05:1**, which is below the 4.5:1 minimum for normal text (WCAG AA). Affects: user chat bubbles and the active media-type button label.
- **Impact**: Users with low vision or in bright ambient conditions will struggle to read text in user chat bubbles and the active media selector button.
- **WCAG**: 1.4.3 Contrast (Minimum) ‚Äî Level AA
- **Recommendation**: Darken the accent to `#7c3aed` (contrast ‚âà 5.25:1 with white ‚úì). Update `--accent` in `:root`, update all `rgba(139, 92, 246, ‚Ä¶)` references across CSS files to `rgba(124, 58, 237, ‚Ä¶)`.
- **Suggested command**: `colorize`

---

#### H-02 ‚Äî Chat message list has no live region

- **Location**: `frontend/src/App.tsx` line 219, `.chat-messages` div
- **Severity**: High
- **Category**: Accessibility
- **Description**: New chat messages (user + assistant) are appended to the DOM but the container has no `aria-live` attribute or equivalent. Screen readers will not announce new messages.
- **Impact**: Blind users running a generation job receive no feedback at all ‚Äî no progress updates, no completion notification, no error announcements.
- **WCAG**: 4.1.3 Status Messages ‚Äî Level AA
- **Recommendation**: Add `role="log"` to the `.chat-messages` div. The `log` role implies `aria-live="polite"` and `aria-relevant="additions text"`, which is semantically correct for a chat timeline.
- **Suggested command**: `harden`

---

#### H-03 ‚Äî Progress bar missing ARIA value attributes

- **Location**: `frontend/src/components/ChatBubble.tsx` lines 32‚Äì38, `.progress-bar-fill` div
- **Severity**: High
- **Category**: Accessibility
- **Description**: The visual progress bar div has no `role="progressbar"`, `aria-valuenow`, `aria-valuemin`, or `aria-valuemax` attributes. It is invisible to screen readers.
- **Impact**: Screen-reader users cannot determine generation progress at all. The only feedback is the text label (`progressLabel`), which is announced by the live region but not associated with the progress bar.
- **WCAG**: 4.1.2 Name, Role, Value (Level A); 1.3.1 Info and Relationships (Level A)
- **Recommendation**: Add `role="progressbar"`, `aria-valuenow={progress}`, `aria-valuemin={0}`, `aria-valuemax={100}`, and `aria-label="Generation progress"` to the `.progress-bar-fill` div.
- **Suggested command**: `harden`

---

#### H-04 ‚Äî Send button touch target below minimum (40√ó40px)

- **Location**: `frontend/src/App.css`, `.composer-send` rule (lines 161‚Äì194)
- **Severity**: High
- **Category**: Accessibility / Responsive
- **Description**: The generate/send button is 40√ó40 CSS pixels. WCAG 2.5.5 requires a minimum touch target of 44√ó44px.
- **Impact**: Users with motor disabilities, or on touch devices, may miss-tap and not trigger generation ‚Äî causing frustration.
- **WCAG**: 2.5.5 Target Size (Level AA)
- **Recommendation**: Increase to `width: 44px; height: 44px`.
- **Suggested command**: `harden`

---

#### H-05 ‚Äî Remove-image button touch target far below minimum (20√ó20px)

- **Location**: `frontend/src/components/ImageUpload.module.css`, `.removeBtn` rule
- **Severity**: High
- **Category**: Accessibility / Responsive
- **Description**: The ‚úï remove-image button is 20√ó20 CSS pixels ‚Äî less than half the WCAG minimum.
- **Impact**: Extremely difficult to tap on mobile. Users with motor disabilities or those using a stylus will frequently miss the target.
- **WCAG**: 2.5.5 Target Size (Level AA)
- **Recommendation**: Keep the visual size at 20√ó20 but expand the interactive hit area to 44√ó44 using `padding: 12px; margin: -12px; box-sizing: content-box`. The negative margin compensates for the layout shift so surrounding elements are unaffected.
- **Suggested command**: `harden`

#### H-06 ‚Äî Download link has no `:focus-visible` style

- **Location**: `frontend/src/components/ChatBubble.module.css`, `.download-btn`
- **Severity**: High
- **Category**: Accessibility
- **Description**: The `.download-btn` anchor element has `:hover` styles but no `:focus-visible` rule. Keyboard users tabbing to the download link receive no visible focus indicator.
- **Impact**: Keyboard-only users cannot see which element is currently focused when tabbing through generated media results, violating the "visible focus indicator" requirement.
- **WCAG**: 2.4.7 Focus Visible (Level AA)
- **Recommendation**: Add `.download-btn:focus-visible { outline: 2px solid var(--accent); outline-offset: 2px; }`.
- **Suggested command**: `harden`

#### M-01 ‚Äî No responsive breakpoints

- **Location**: `frontend/src/App.css` ‚Äî no `@media` queries defined anywhere
- **Severity**: Medium
- **Category**: Responsive
- **Description**: The layout uses `max-width: 900px` for desktop but has zero `@media` queries. On narrow viewports (‚â§ 480px) the composer config row with MediaTypeSelector + ParameterPanel may overflow or wrap awkwardly.
- **Impact**: Mobile users may encounter crowded or overflowing UI at small widths.

#### M-02 ‚Äî `field-sizing: content` has limited browser support

- **Location**: `frontend/src/App.css`, `.composer-textarea` (line 148)
- **Severity**: Medium
- **Category**: Performance / Compatibility
- **Description**: `field-sizing: content` is a cutting-edge CSS property (Chromium 123+, Firefox 129+, Safari 17.4+). Older browsers do not auto-grow the textarea.
- **Impact**: Users on Safari < 17.4 or Firefox < 129 see a single-row textarea regardless of input length, degrading the prompt UX.
- **Recommendation**: Add a JS resize handler as a progressive enhancement fallback: listen to the `input` event and set `element.style.height = 'auto'; element.style.height = element.scrollHeight + 'px'`.
- **Suggested command**: `harden`

---

#### M-03 ‚Äî Generated images missing `loading` attribute and dimensions

- **Location**: `frontend/src/components/ChatBubble.tsx` line 55
- **Severity**: Medium
- **Category**: Performance
- **Description**: The generated `<img>` element has no `loading="lazy"` and no explicit `width`/`height` attributes.
- **Impact**: The browser cannot reserve layout space, causing cumulative layout shift (CLS). On a slow connection, the image loads eagerly even if off-screen.
- **Recommendation**: Add `loading="lazy"` and `decoding="async"` to the generated image. Dimensions may not be known at render time ‚Äî consider rendering with `width: 100%` in CSS (already done) and accept dynamic height.
- **Suggested command**: `optimize`

#### M-04 ‚Äî Video and audio elements missing accessible labels

- **Location**: `frontend/src/components/ChatBubble.tsx` lines 63‚Äì76
- **Severity**: Medium
- **Category**: Accessibility
- **Description**: The `<video>` and `<audio>` elements have no `title` or `aria-label` attribute. Screen readers will announce them with no context.
- **Impact**: Screen-reader users hear "video" or "audio" with no indication of what was generated.
- **WCAG**: 4.1.2 Name, Role, Value (Level A)
- **Recommendation**: Add `aria-label="Generated video"` / `aria-label="Generated audio"` and `title` attributes. Ideally include the prompt text.
- **Suggested command**: `harden`

#### M-05 ‚Äî Error and connection-lost states use colour-only differentiation

- **Location**: `frontend/src/components/ChatBubble.module.css`, `.status-error` and `.status-connection-lost`
- **Severity**: Medium
- **Category**: Accessibility
- **Description**: Error states change text colour only (`#fca5a5` for error, `#fcd34d` for connection-lost). No icon, no prefix text, no pattern. Users who cannot distinguish colours have no additional signal.
- **Impact**: Colour-blind users may not distinguish a failed generation from a normal message.
- **WCAG**: 1.4.1 Use of Color (Level A)
- **Recommendation**: Prefix error content with a symbolic indicator (e.g. `‚öÝ ` for connection-lost, `‚úï ` for error) rendered via the component or via CSS `::before` content.
- **Suggested command**: `harden`

#### M-06 ‚Äî No skip-to-main-content link

- **Location**: `frontend/index.html` / `frontend/src/App.tsx`
- **Severity**: Medium
- **Category**: Accessibility
- **Description**: There is no skip-navigation link at the top of the page. Keyboard users must tab through the header and entire composer before reaching the chat area.
- **Impact**: Users who rely on keyboard navigation (sighted and blind alike) face unnecessary friction on every page load.
- **WCAG**: 2.4.1 Bypass Blocks (Level A)
- **Recommendation**: Add a visually-hidden `<a href="#chat-area" className="skip-link">Skip to main content</a>` that becomes visible on focus, and add `id="chat-area"` to `<main>`.
- **Suggested command**: `harden`

## Patterns & Systemic Issues

1. **Hardcoded RGBA values throughout CSS files** ‚Äî `rgba(139, 92, 246, ‚Ä¶)` appears in 4 separate CSS files instead of using `rgba(var(--accent-rgb), ‚Ä¶)`. A change to the accent colour required updating 10+ individual declarations manually.
2. **Touch targets consistently too small** ‚Äî both the send button (40px) and remove-image button (20px) fall below the 44px minimum. No other interactive elements were audited for touch target size (ParameterPanel selects/inputs may also need checking on mobile).
3. **Missing ARIA attributes pattern** ‚Äî interactive live-region patterns (progress, chat log, status messages) are all missing ARIA enrichment. These are systematic gaps, not one-off mistakes.
