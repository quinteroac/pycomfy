# Binary Build Pipeline: PyInstaller + GitHub Actions CI

## Context

End users (non-developers) need to install the `parallax` CLI without Python or any prior toolchain. This PRD covers compiling the CLI into standalone executables for Linux (x86_64), macOS (universal binary: arm64 + x86_64), and Windows (x86_64) using PyInstaller, and publishing those binaries as assets on GitHub Releases via a CI workflow triggered on version tags. The stack is 100% Python — no Bun or Node.js build steps are involved.

## Goals

- Produce a single-file `parallax` executable per supported platform that requires no runtime dependencies.
- Automate the build and publish process so every tagged release produces ready-to-download binaries with zero manual steps.
- Keep binary size reasonable by excluding heavy inference deps (torch, transformers, comfy_diffusion) — the installer subcommands pull those at runtime via uv.

## User Stories

### US-001: Build standalone binary locally
**As a** maintainer, **I want** to run a single command locally **so that** I can produce a `parallax` binary for my current platform and test it before tagging a release.

**Acceptance Criteria:**
- [ ] Running `uv run pyinstaller parallax.spec` produces a single-file executable at `dist/parallax` on Linux/macOS or `dist/parallax.exe` on Windows.
- [ ] The produced binary runs `parallax --help` and `parallax install --help` without any Python installation present on the machine.
- [ ] The binary does not bundle `torch`, `torchvision`, `torchaudio`, `transformers`, or `comfy_diffusion` — they are listed in `excludes` in the `.spec` file.
- [ ] The binary size is under 50 MB.

---

### US-002: CI builds binaries on every version tag
**As a** maintainer, **I want** GitHub Actions to automatically build and publish platform binaries whenever I push a `v*.*.*` tag **so that** users always have up-to-date downloads without manual steps.

**Acceptance Criteria:**
- [ ] A workflow file `.github/workflows/release-cli.yml` triggers on `push` to tags matching `v*.*.*` and on `workflow_dispatch`.
- [ ] The workflow runs three parallel jobs: `linux` on `ubuntu-latest`, `macos` on `macos-latest` (produces a universal2 binary), and `windows` on `windows-latest`.
- [ ] Each job: checks out the repo, installs uv, installs only the CLI build deps (`uv sync --group cli-build --no-group dev`), runs PyInstaller with `parallax.spec`, and produces the binary.
- [ ] The macOS job compiles a universal binary using `--target-arch universal2` so it runs natively on both Apple Silicon and Intel Macs.
- [ ] Each job generates a SHA256 checksum file alongside the binary (`parallax-linux-x86_64.sha256`, `parallax-macos-universal.sha256`, `parallax-windows-x86_64.exe.sha256`).
- [ ] After all three build jobs succeed, a final `release` job uses `softprops/action-gh-release` to create or update the GitHub Release for the tag and uploads all 6 files (3 binaries + 3 checksums) as release assets.
- [ ] If any build job fails, the release job does not run.

---

### US-003: Binary is self-identified
**As a** user, **I want** `parallax --version` to print the version matching the GitHub Release tag **so that** I know exactly which binary I am running.

**Acceptance Criteria:**
- [ ] `parallax --version` prints `parallax 1.x.x` where `1.x.x` matches the `[project].version` field in `pyproject.toml` at the time of the build.
- [ ] The version is baked into the binary at build time via a build-time constant (not read from an installed package at runtime, since comfy-diffusion is not bundled).

---

## Functional Requirements

- FR-1: A `parallax.spec` PyInstaller spec file is added at the repo root. It sets `onefile=True`, `console=True`, `name="parallax"`, and lists `torch`, `torchvision`, `torchaudio`, `transformers`, `comfy`, `comfy_diffusion`, `cv2`, `numpy`, `PIL` in `excludes`.
- FR-2: A `[project.optional-dependencies]` group `cli-build` is added to `pyproject.toml` containing `pyinstaller>=6.0` so the build dep is trackable without polluting the main deps.
- FR-3: The CI `macos` job passes `--target-arch universal2` to PyInstaller and produces a single `parallax-macos-universal` asset (not two separate arm64/x86_64 files).
- FR-4: Release asset names follow the convention `parallax-{platform}[.exe]` exactly: `parallax-linux-x86_64`, `parallax-macos-universal`, `parallax-windows-x86_64.exe` — the bootstrap script (PRD-003) constructs download URLs from these names.
- FR-5: SHA256 checksums are generated with `sha256sum` (Linux/macOS) or `CertUtil` (Windows) and written to `{asset-name}.sha256` as a single line: `<hex>  <filename>`.
- FR-6: The version constant baked into the binary is read from `pyproject.toml` at build time via a `__version__` module generated by the `.spec` file's `Analysis` step.

## Non-Goals

- Code-signing or notarization of the macOS binary (deferred to a future iteration).
- Linux ARM support (deferred).
- Publishing to Homebrew tap or winget (deferred).
- Implementing installer subcommands (covered in PRD-001).
- Writing the bootstrap install script (covered in PRD-003).
